VPC Lambda + Secrets Manager (ConnectTimeout to secretsmanager.*.amazonaws.com)
================================================================================

If a Lambda is attached to VPC subnets that have NO path to the internet, calls to
AWS Secrets Manager (public HTTPS endpoint) will time out.

Typical fixes (pick one):

1) Interface VPC endpoint (recommended for private-only subnets)
   - Service name: com.amazonaws.<region>.secretsmanager
   - Type: Interface
   - Subnets: same subnets as the Lambda (or subnets routed to the Lambda SG)
   - Security group: allow TCP 443 inbound from the Lambda security group
   - Private DNS enabled on the endpoint (usually default)
   - If private DNS is OFF: set Lambda env SECRETS_MANAGER_ENDPOINT_URL to the endpoint DNS
     (HTTPS URL for the VPC endpoint, from the console), so boto3 does not use the public URL.

2) NAT Gateway
   - Private subnet route table: 0.0.0.0/0 -> NAT Gateway
   - Lambda in private subnets uses NAT for outbound HTTPS to Secrets Manager

3) Temporary / emergency: avoid Secrets Manager API at runtime
   - Set on the Lambda: RDS_USE_ENV_CREDENTIALS=true
   - And set RDS_ENDPOINT, RDS_USERNAME, RDS_PASSWORD, RDS_DATABASE (+ optional RDS_PORT, RDS_SSLMODE)
   - Prefer fixing VPC networking and using Secrets Manager long term.

Also verify RDS_SECRET_ARN points at a *Secrets Manager secret ARN* (secretsmanager:...:secret:...),
not an RDS cluster/instance ARN.
